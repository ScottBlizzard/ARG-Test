## Slide 4. 工具工件与 Prompt 设计

**发言人：** 3号成员  
**建议时长：** 60-75 秒

**中文理解版**

在正式实验中，我们使用的核心模型是 Qwen3.5-Flash。

我们不是只用一个 prompt，而是采用了一组 prompt family。

其中，system prompt 用来定义测试助手的整体行为；generation prompt 用来要求模型输出结构化的推理轨迹；repair prompt 用来根据 checker 的反馈修补较弱或失败的输出；此外我们还保留了一个 plain baseline prompt 用来做比较。

这一点很重要，因为它体现了团队和 AI 的交互方式。最终结果不是来自一次简单的模型调用，而是来自团队设计并不断修订的提示词策略。

## Slide 5. 流程与架构

**发言人：** 4号成员  
**建议时长：** 60-75 秒

**中文理解版**

这一页展示的是 ARG-Test 的完整流程。

整个流程包含六个主要阶段。第一步，从 requirement 生成结构化 trace。第二步，把模型输出解析成类型化表示。第三步，运行 checker suite 做契约验证。第四步，对多个候选结果做 reranking。第五步，在必要时进入 repair loop。最后一步，把结果导出为最终测试工件。

所以我们的系统不是简单的“输入一个 prompt，输出一个答案”，而是一条包含生成、检查、筛选和修补的完整流程。

## 简短 Q&A 备用题库

### Q1. 为什么你们选择 requirement 作为输入？

因为 requirement-driven 的黑盒测试完全符合老师作业要求，而且它和结构化推理、coverage 导向评估天然契合。

### Q2. 为什么你们的方法比 plain prompting 更好？

因为我们不依赖单次 prompt 输出。我们把结构化推理、解析、检查、reranking 和 repair 放进了整个系统里，从而提升可靠性和可审计性。

### Q3. checker 能证明测试用例完全正确吗？

不能。checker 不是形式化证明器。它的作用是检查生成的推理轨迹和测试套件是否满足关键测试义务和契约。

### Q4. 团队的主要贡献是什么？

团队设计了 prompts、pipeline、checking logic、dataset、实验设置，以及最终的修订策略。AI 只是整个工作流的一部分，而不是全部。

### Q5. 工具到底在哪里？你们这真的是一个工具吗，还是只是一个测试想法？

是的，这是真正的工具，而不只是一个测试想法。这个工具就是 ARG-Test 的整个 pipeline。它接收 requirement 文件作为输入，用结构化 prompts 调用模型，解析响应，用契约型 validator 做检查，对候选结果 rerank，在必要时 repair，最后导出结构化测试套件和评估报告。所以这里的“工具”是整条可运行的 workflow，而不是最后那次实验或者单个 prompt。

### Q6. 你们工具的输入和输出分别是什么？

当前实现里，主要输入是自然语言 requirement。输出是结构化的黑盒测试套件，以及中间和最终工件，比如 reasoning trace、checker results、coverage report 和导出的 test cases。也就是说，系统不只是生成文字，而是生成可以被检查和评估的测试工件。

### Q7. 你们项目里是怎么定义一个 test case 的？

在我们的项目里，一个 test case 被定义为一个结构化的测试单元，它来自一个或多个 requirement item。最基本地，它至少包含具体输入或场景、预期输出或预期系统行为，以及它要覆盖的 requirement item 或测试义务。很多情况下，我们还会记录测试技术，比如等价类划分或边界值分析，以便后续审计。

### Q8. 这些 test cases 是基于什么定义或推导出来的？

我们是基于经典黑盒测试原理以及 requirement 自身的约束来定义和推导 test case 的。具体包括等价类划分、边界值分析、决策规则推理，以及在适用时的 workflow 或状态转换推理。所以模型不是在随意生成 case，而是在测试理论、requirement 规则和 checker contract 的共同约束下生成。

### Q9. 为什么你们没有实现 codebase-input 分支？

因为老师的作业允许从 system requirements 或 testing codebase 二选一。我们选择 requirement-driven 分支，是因为它和黑盒测试以及参考论文里的结构化推理思想更契合。所以我们选择把一条分支做好，而不是两条分支都做得很浅。未来可以通过加入 code-to-requirement 或 code-to-contract 的适配层，把同一框架扩展到 codebase input。

### Q10. 为什么选 Qwen3.5-Flash？它可靠吗？

我们选择 Qwen3.5-Flash，是因为它在课程项目场景下在成本、速度和输出质量之间提供了一个比较平衡的选择。更重要的是，我们的项目并不是直接信任模型原始输出。我们把模型包在一个结构化的 pipeline 里，外面还有 parsing、checking、reranking 和 repair。所以最终工件的可靠性来自整个系统设计，而不是假设底层模型本身永远正确。

### Q11. 你们怎么知道 checker 是有效的？

我们的 checker 不是一个完整的语义证明引擎。它的目标更窄，也更实际：验证生成的 reasoning trace 和 final test suite 是否满足明确的测试义务。这些义务来自标准黑盒测试逻辑，比如覆盖有效/无效等价类、边界点、决策规则和工作流转换。所以 checker 作为一个基于 contract 的验证层是有效的，虽然它不能保证完整的真实世界正确性。

### Q12. 你们的 average coverage 只有 0.615，为什么还能说项目成功？

因为我们项目的目标不是让每条 requirement 都达到完美 coverage，而是构建一个结构化、可审计，并且明显优于简单 baseline 的 AI 增强测试工具。从这个意义上说，项目是成功的，因为 full pipeline 明显优于 rule-based 和 plain-prompt baseline，同时还给出了较高的 checker score 和稳定的结构化工件。所以这里的“成功”指的是显著改进和完整可运行的工具，而不是完美 coverage。

### Q13. 你们和 baseline 的比较公平吗？

是公平的。因为所有 baseline 都是在同一个 requirement-to-test generation 任务上、同一个数据集上进行比较。rule-based baseline 代表传统非 AI 方法，plain LLM baseline 代表没有结构控制的直接提示，structured-no-checker baseline 用来隔离“结构化”本身的作用，而 full pipeline 则体现 checking、reranking 和 repair 的额外价值。这样的设计既能比较最终性能，也能比较各个组件的贡献。

### Q14. 你们当前项目最大的局限性是什么？

最大的局限性是，我们当前更聚焦 requirement-driven testing 和结构化生成质量，而不是在一个大型真实软件系统上做完全可执行的端到端验证。另外，我们的数据集规模仍然是课程项目级别，而不是工业级规模。所以我们的结论应该被理解为：在这个设定下，方法是有效的、有力的证据，而不是在所有测试场景下都具有普适完备性。

### Q15. 如果模型输出错了，你们系统会怎么做？

如果模型原始输出较弱、不一致或不完整，系统不会直接接受它。相反，它会先解析输出，再检查关键测试义务有没有缺失，然后比较多个候选结果，并在必要时进入 repair 阶段。也就是说，系统内置了一套检测和缓解弱生成结果的机制，而不是盲目相信模型的第一次回答。
