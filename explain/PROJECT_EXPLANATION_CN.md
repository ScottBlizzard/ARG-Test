# ARG-Test 项目通俗解释

## 这个项目一句话是干什么的

这个项目可以把一段**自然语言写的需求**，自动变成一套**测试用例设计结果**。

你可以把它理解成一个“AI 辅助测试设计工具”：

- 输入：一段 requirement
- 输出：一套结构化测试用例
- 额外能力：会检查这些测试是不是像样、是不是漏了边界、是不是漏了非法情况、是不是能导出成报告

它不只是“让大模型写几个测试点”，而是想做成一个更正规的流程。

## 这个项目为什么要做

普通大模型有一个问题：

- 它很会说
- 但不一定说得完整
- 也不一定方便检查

比如你让它根据需求写测试，它可能会写：

- 测一个正常情况
- 测一个错误情况
- 测一个边界

看起来没问题，但你很难知道：

- 它是不是漏了重要规则
- 它是不是真的覆盖了边界
- 它有没有考虑非法输入
- 它说自己用了某种测试方法，到底是不是真的用了

所以这个项目的目的就是：

**不要直接相信 LLM 的结果，而是把 LLM 产出的测试设计过程变得可检查、可追踪、可复现。**

## 你可以把它想成什么

可以把它想成一个“测试设计流水线”。

不是你问一句，大模型回一句就结束了。

而是：

1. 先让模型按规定格式输出
2. 再把输出拆开解析
3. 再检查有没有问题
4. 如果不够完整，就补一补
5. 最后再导出成正式结果

所以它更像一个“小型自动测试设计工厂”，不是一个聊天机器人。

## 它到底处理什么输入

它主要处理的是**需求文本**，比如：

- 优惠券规则
- 表单校验规则
- 状态流转规则
- 订单审批流程

仓库里这些需求放在：

- `data/requirements/dev/`
- `data/requirements/test/`

也就是说，这个项目主要是在做：

**从需求出发的黑盒测试设计**

不是做代码生成，也不是做静态代码分析。

## 它最终输出什么

它最终会输出一套结构化测试结果，里面通常会有：

- 测试编号
- 使用了什么测试技术
- 输入是什么
- 预期输出是什么
- 覆盖了哪条规则
- 优先级是什么

还能导出成：

- `JSON`
- `CSV`
- `Markdown`

这些最终结果一般会出现在：

- `outputs/final_tests/`
- `outputs/reports/`

## 它和普通“让 AI 写测试”最大的区别是什么

最大区别是：

**它要求 AI 先按固定结构思考，再让程序去检查这个思考过程。**

项目里要求模型输出五个固定部分：

1. `Analysis`
2. `Pattern`
3. `Steps`
4. `Verification`
5. `FinalAnswer`

意思分别可以通俗理解成：

- `Analysis`：先分析需求里有什么规则
- `Pattern`：决定用哪些测试方法
- `Steps`：说明测试是怎么一步步设计出来的
- `Verification`：自己检查有没有漏
- `FinalAnswer`：最后给出正式测试表

这样一来，结果就不再是“黑箱”了。

## 一个需求进来以后，系统大概怎么跑

可以把流程理解成下面这样：

### 第一步：读入需求

它可以从几种地方读需求：

- 读一个 `.txt` 文件
- 直接输入一段文本
- 从 CSV 批量导入

这部分主要是 `src/main.py` 和 `src/input_loader.py` 在做。

### 第二步：让模型生成多个版本

它不会只让模型生成一次，而是会生成多个 candidate。

原因很简单：

- 大模型每次输出可能不一样
- 有的版本更重视边界
- 有的版本更重视负例
- 有的版本更平衡

所以项目会先多生成几个版本，再从里面挑一个最好的。

这部分主要在 `src/pipeline.py` 和 `src/llm_client.py`。

### 第三步：把模型输出拆开

模型输出回来以后，程序会检查：

- 有没有 `Analysis`
- 有没有 `Pattern`
- 有没有 `Steps`
- 有没有 `Verification`
- 有没有 `FinalAnswer`

还会把 `FinalAnswer` 里的表格解析成程序能处理的测试用例对象。

这部分在 `src/parser.py`。

### 第四步：检查它写得对不对、全不全

这是这个项目很重要的一步。

程序会自动检查：

- 有没有正常情况和异常情况
- 有没有边界值
- 有没有规则组合测试
- 如果是状态流转需求，有没有合法和非法迁移

这部分是 `src/checker/` 目录下的一组 checker 在做。

可以理解成：

**模型先写答案，checker 再像助教一样检查作业。**

### 第五步：如果不够完整，就自动补

如果 checker 发现缺东西，比如：

- 没写 invalid case
- 没写边界值
- 没写 illegal transition

系统会做一个轻量 repair，把缺的东西补进去。

这部分在 `src/repair.py`。

### 第六步：给结果打分、排序、选最好的

多个 candidate 都检查完以后，系统会综合考虑：

- checker 分数
- 有没有重复测试
- 技术覆盖是不是更丰富

然后挑一个最好的结果。

这部分在 `src/reranker.py`。

### 第七步：补充风险和状态模型信息

项目不只是出测试用例，还会额外补两类信息：

- `risk`：这个需求风险高不高
- `state model`：如果这是流程类需求，它有哪些状态和状态迁移

这两部分分别在：

- `src/risk.py`
- `src/state_model.py`

### 第八步：拿 gold spec 做评估

项目里还有一套人工整理好的“标准答案参考”，叫 `gold spec`。

它不是拿来给模型看的，而是拿来做评估的。

系统会比较：

- valid partition 覆盖了多少
- invalid partition 覆盖了多少
- boundary 覆盖了多少
- decision rule 覆盖了多少
- state / illegal transition 覆盖了多少

这部分在 `src/evaluation/metrics.py`。

## 风险评估是干什么的

项目不是只想生成测试，还想告诉你：

**哪些需求更值得优先测。**

比如下面这些需求通常更高风险：

- 有金额、支付、优惠券
- 有严格拒绝条件
- 有很多规则组合
- 有复杂流程和状态迁移

系统会根据这些线索打一个 risk score，然后给出：

- `Low`
- `Medium`
- `High`

还会顺带给每条测试用例调 priority。

简单说就是：

**不仅帮你设计测试，还帮你判断先测什么。**

## 状态模型那部分是干什么的

如果 requirement 是流程类的，比如：

- 订单从 `Draft` 到 `Submitted`
- 审批成功后进入另一个状态
- 某些状态之间禁止跳转

项目会尝试从文本里提取：

- 有哪些状态
- 哪些迁移是合法的
- 哪些迁移是非法的
- 怎么设计序列去覆盖这些状态和迁移

这就是 `state model` 那部分。

所以这个项目不只是会做输入校验类测试，也能做一部分 workflow / state transition 类测试设计。

## demo_web 是什么

`demo_web/` 不是另一个核心算法项目，它更像一个展示界面。

它的作用是让人更方便地现场演示：

- 直接输入需求
- 上传 CSV
- 查看状态模型
- 查看 formal evidence

也就是说：

- `src/` 是发动机
- `demo_web/` 是仪表盘

## experiments 是什么

`experiments/` 是实验脚本目录。

它不是核心算法本身，但很重要，因为你们要拿它证明项目真的有效。

这里面大概有几类脚本：

- `run_main.py`
  跑主流程
- `run_baselines.py`
  跑几个 baseline 做对比
- `run_ablation.py`
  做消融实验
- `run_generalization.py`
  看不同类别 requirement 上的表现
- `run_repeatability.py`
  看重复运行稳不稳定
- `run_nfr_checks.py`
  做性能、可用性、安全、可维护性检查
- `run_mutation_demo.py`
  做 mutation usefulness 演示

所以 `experiments/` 可以理解成：

**证明这个工具不是“看起来能跑”，而是真的经过验证。**

## tests 是什么

`tests/` 是 pytest 测试。

里面测的不只是一个点，而是几层内容：

- Web API 能不能正常工作
- CSV 输入和 state model 能不能跑
- risk 评估逻辑是不是合理
- reproducibility 控制有没有生效
- coupon 模块黑盒/白盒测试是不是通过
- mutation demo 能不能杀死 mutant

也就是说，这个仓库不是只做“项目展示”，而是确实有代码级测试。

## coupon_discount_engine 是什么

这个很容易混淆。

`coupon_discount_engine` **不是整个项目本体**。

它是你们为了做 detailed execution，专门选出来的一个**可执行参考模块**。

它的作用是：

- 给项目找一个可以真正跑起来的目标模块
- 让你们能做黑盒测试
- 让你们能做白盒测试
- 让你们能做 coverage
- 让你们能做 mutation demo

也就是说：

- `ARG-Test` 是测试设计工具
- `coupon_discount_engine` 是被你们拿来做执行验证的参考模块

这两者不是一回事。

## 这个项目和普通脚本相比，真正有价值的地方在哪里

我觉得它的价值主要在这几层：

- 它不是直接让 LLM 瞎写，而是强制结构化
- 它不是只生成结果，还会检查结果
- 它不是只检查，还会做修补
- 它不是只导出测试，还会做风险评估
- 它不是只讲概念，还做了 baseline、ablation、repeatability、NFR、mutation 这些验证

所以它最重要的特点不是“用了 AI”，而是：

**把 AI 生成测试这件事，做成了一个更像正规软件测试工具的流程。**

## 用最通俗的话总结

如果完全不说术语，这个项目可以这样理解：

> 你给它一段软件需求，它会先让 AI 按规定格式分析这段需求，再自动检查 AI 有没有漏掉正常情况、异常情况、边界值、规则组合和状态迁移；如果漏了，它会补；最后再把结果整理成正式测试用例，并用一套参考标准去评估质量。

再短一点：

> 这是一个“把需求自动变成测试设计，并且尽量让这个过程可检查、可信一点”的项目。
